# 곡 드래그 배치 계약 — 2026-07-22

## 1. 사용자 목표

검색 결과와 가져온 노래책의 **재생 가능한 곡**을 끌어 다음 세 위치에 놓을 수 있게 한다.

1. 지금/다음 재생
2. 대기열 끝
3. 이전 재생곡

클릭은 기존의 `곡 정보 확인` 화면을 연다. 드래그는 반복 작업을 줄이는 빠른 경로이며,
클릭 경로를 없애거나 모바일·키보드 사용자를 드래그에 종속시키지 않는다.

## 2. 상태 모델

| 상태 | 화면 | 허용 전이 | 금지 사항 |
|---|---|---|---|
| `idle` | 평상시 곡 목록 | 재생 가능한 곡의 drag start → `dragging` | 상태·대기열·이력 변경 |
| `dragging` | 하단에 세 목적지 표시 | 목적지 진입 → `over`; drag end → `idle` | 서버 명령, 준비 요청, 목록 변경 |
| `over(destination)` | 가리킨 목적지만 강조 | drop → `committing`; 이탈 → `dragging` | 다른 목적지 동시 강조 |
| `committing` | 선택한 목적지에 한 번 반영 | 성공/실패 안내 후 `idle` | 중복 반영, drag end 재반영 |
| `waiting-to-play(entryId)` | Speaker의 정확한 곡 한 건이 대기열 맨 앞에서 준비 중 | 같은 항목의 `ready` → `starting`; 현재 곡·출력·대기열 변경 → `idle` | 다른 항목 시작, OBS 지연 자동 시작, reload 뒤 의도 복원 |

드래그 후보는 페이지 메모리에만 둔다. `localStorage`, Worker, WebSocket에는 drop 전까지
어떤 메시지도 보내지 않는다.

`waiting-to-play` 역시 현재 탭 메모리에만 둔다. 대기열 데이터는 기존처럼 저장하지만
“준비가 끝나면 재생”이라는 한 번의 사용자 의도는 reload·다른 탭·출력 전환을 넘어
되살리지 않는다. 의도는 `entryId + src + 요청 당시 출력`으로 식별하며, 대기열 첫 항목이
달라지면 조용히 폐기한다.

## 3. 목적지 계약

### A. 지금/다음 재생

- 현재 곡이 없고 곡 준비가 끝났으면 기존 재생 시작 경로를 그대로 사용한다.
- 현재 곡이 있으면 **현재 곡을 즉시 잘라 버리지 않고** 대기열 맨 앞으로 둔다.
- Speaker에서 현재 곡이 없고 YouTube 곡이 준비 중이면 정확한 새 `entryId`를 대기열
  맨 앞에 두고 `waiting-to-play`를 연다. 같은 항목이 `ready`가 된 때에만 한 번 시작한다.
- 기다리는 동안 다른 곡이 시작되거나, 출력이 바뀌거나, 항목이 제거·재정렬되면 자동
  시작 의도를 폐기한다. 곡 자체는 사용자가 바꾼 대기열 위치에 그대로 남긴다.
- OBS에서 준비 중인 곡은 대기열 맨 앞에만 둔다. 준비 완료나 연결 회복이 나중에
  도착했다는 이유로 방송 오디오를 자동 시작하지 않으며 사용자가 직접 재생한다.
- `blocked`·영구 `unavailable`은 기존 규칙대로 거절한다. 일시적인 준비/연결 지연은
  Speaker의 의도를 유지하되 실제 `ready` 증거 전에는 run을 만들지 않는다.
- OBS에서는 기존 route/lease/output safety 게이트를 우회하지 않는다.

### B. 대기열 끝

- 현재 곡 유무와 무관하게 새 `entryId`를 발급해 대기열 끝에 둔다.
- 준비 중인 곡도 예약할 수 있지만, 영구 unavailable/blocked 곡은 기존 규칙대로 거절한다.
- 같은 곡을 여러 번 놓으면 서로 다른 신청으로 취급한다.

### C. 이전 재생곡

- 미디어 재생이나 OBS 명령 없이 새 completed 항목을 이력 끝에 둔다.
- 실제로 재생했다는 시스템 증거로 오해하지 않도록 `completionReason`을 만들지 않는다.
- 이력의 `다시 예약` 동작은 기존처럼 새 `entryId`의 대기열 항목을 만든다.

## 4. 드래그 가능한 곡

- YouTube 검색 결과: 영상 ID가 있는 모든 결과.
- YouTube 플레이리스트/Setlink/멜로밍 노래책: 곡명 준비가 끝났고, 캐시 MR 또는 목록의
  YouTube MR ID가 실제로 있는 항목만.
- MR이 없는 노래책 곡, 제목 정리 중인 곡, 로컬 파일 선택 상자는 드래그 후보가 아니다.
  이들은 클릭/파일 선택으로 기존 검토 흐름을 사용한다.

## 5. 접근성·편의성

- 곡 행 전체의 클릭 행동은 바꾸지 않는다.
- 드래그 가능한 행에는 마우스 사용자가 발견할 수 있는 손잡이와 번역된 도움말을 둔다.
- 목적지 이름은 실제 결과에 맞춰 `지금 재생`, `준비 후 재생`, `다음 재생`,
  `방송 대기` 중 하나로 정직하게 바뀐다.
- 드래그 목적지는 `aria-hidden`인 포인터 보조 UI다. 키보드와 터치 환경의 정식 경로는
  클릭 → 검토 → 재생/대기열 버튼이다.
- 새 사용자 문구는 모두 `appMessages`의 한국어/영어 동일 키로 관리한다.

## 6. 성능 경계

- 드래그 트레이는 drag 중에만 DOM에 생성하고 평상시 DOM·이벤트 비용은 거의 0으로 둔다.
- 새 외부 의존성, 이미지, 네트워크 요청을 추가하지 않는다.
- drop 한 번은 상태 갱신 한 번과 기존 준비/재생 호출만 허용한다.
- 1,000곡 이력의 고정 100행 렌더 계약과 Blob 예산은 변경하지 않는다.

## 7. 검증 관문

1. 검색 결과 클릭은 계속 `곡 정보 확인`으로 간다.
2. 검색 결과를 세 목적지에 각각 놓았을 때 정확히 한 위치만 바뀐다.
3. 현재 곡이 있을 때 재생 목적지는 현재 곡을 즉시 버리지 않고 큐 맨 앞에 둔다.
4. MR 없는 노래책 곡은 draggable이 아니며 클릭하면 MR 찾기 흐름으로 간다.
5. MR 연결 곡은 세 목적지에 놓을 수 있고 source/songbookId/tags를 보존한다.
6. 취소한 드래그는 state, localStorage, Worker 요청을 만들지 않는다.
7. 320/375/768/1100px에서 가로 overflow가 없고 트레이의 세 목적지가 잘리지 않는다.
8. 한국어/영어 전환 즉시 트레이와 도움말이 함께 바뀐다.
9. 기존 대기열/이력 entryId 드래그, 로컬 파일 drop과 타입이 섞이지 않는다.
10. 전체 단위 테스트, 브라우저 smoke, OBS 번들 예산, 500회 Speaker↔OBS 전환을 다시 통과한다.
11. Speaker의 준비 지연 곡은 같은 `entryId`가 ready일 때 정확히 한 번 시작한다.
12. 기다리는 사이 현재 곡·출력·대기열 첫 항목이 바뀌면 자동 시작하지 않는다.
13. OBS의 준비 지연·route 미확정 drop은 대기열 맨 앞에 남고 자동 방송하지 않는다.
