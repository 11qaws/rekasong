# Rekasong 현재 상태와 다음 작업

> 기준일: 2026-07-16
> 작성 방식: 현재 소스 기준 정적 검토 + 로컬 프로덕션 빌드 확인

이 문서는 현재 저장소의 구현 상태, 이번 기록에 포함된 변경, 그리고 [곡 생애주기와 상태 모델](SONG_LIFECYCLE.md)을 충족하기 위해 남은 작업을 기록한다.

## 1. 제품의 현재 목표

Rekasong은 스트리머가 미리 검증한 “부를 수 있는 곡 + 바로 재생 가능한 MR”을 카탈로그로 관리하고, 방송 중 요청곡을 확인한 뒤 명시적으로 대기열·재생으로 전환하도록 돕는 도구다.

안전 원칙은 다음과 같다.

- 자동 요청곡 추가·자동 재생은 하지 않는다.
- MR URL/캐시가 있다는 것만으로 실제 방송 준비 완료라고 말하지 않는다.
- 현재 곡의 스킵은 정상 종료 경로를, 쓰레기통은 폐기 경로를 사용한다.
- 실제 OBS Player가 확인하기 전에는 화면·위젯이 재생/종료를 단정하지 않는다.

## 2. 현재 구현되어 있는 범위

### 입력·카탈로그

- YouTube 검색·URL 입력과 YouTube 플레이리스트 입력 경로가 있다.
- 멜로밍과 Setlink를 노래책 입력원으로 불러오는 경로가 있다.
- 로컬 오디오/MP4를 스테이징하고, On-Air 모드에서는 임시 업로드 자산으로 전환하는 경로가 있다.
- AI 제목 정리, 제목 캐시, 노래책 MR 매핑 캐시가 있다.
- 노래책에 이미 등록된 제목은 AI 정리 없이 그대로 사용하는 흐름이 있다.

### 방송 제어·표시

- 대시보드에 현재 재생, 대기열, 이전 재생, 곡 스테이징 화면이 있다.
- 직접 재생 모드에서는 YouTube iframe과 로컬 audio/video 요소를 재생한다.
- On-Air 모드에서는 Cloudflare Durable Object Worker, R2 임시 자산, OBS Player 위젯, 화면정보 위젯을 사용한다.
- OBS 설정은 화면정보 위젯과 On-Air Player URL을 분리해 제공한다.
- 방송 세션 종료 시 Worker가 임시 자산 삭제를 예약하는 구조가 있다.

### 이번 기록에 포함한 변경

- [곡 생애주기와 상태 모델](SONG_LIFECYCLE.md)을 프로젝트의 규범적 설계 기준으로 추가했다.
- UI/UX의 명시성을 “생애주기 전이를 사용자가 정확히 인식·조작할 수 있게 하는 것”으로 문서화했다.
- README에 생애주기 문서와 현재 상태 문서의 링크를 추가했다.
- 개념이 확정되지 않은 비상 정지 버튼은 현재 재생 UI에서 비노출로 바꿨다.
- 직접 재생에서 소스가 비워질 때 재생 상태도 함께 `false`가 되도록 정리했다.

## 3. 현재 구현과 목표 상태의 차이

현재 재생 구조는 생애주기 모델로 옮겨가는 과도기다. 아래 항목은 실제 방송에 쓰기 전에 우선 해소해야 한다.

### P0 — 방송 안전과 상태 진실성

- [ ] **Worker를 유일한 재생 진실원으로 전환**
  현재는 Dashboard React state, Worker transport, 화면정보 위젯 상태가 따로 움직인다. Dashboard의 낙관적 상태 변경이 실제 Player 이벤트보다 앞서면 방송 출력과 화면이 달라질 수 있다.
- [ ] **`QueueEntry`와 `PlaybackRun` 도입**
  현재는 `currentSong/queue/history`와 곡 ID를 사실상 재생 세션 ID로 함께 사용한다. `entryId + runId + playerInstanceId`로 명령과 이벤트를 검증해야 늦은 ended 이벤트가 다음 곡을 끝내지 않는다.
- [ ] **스킵을 `finish → ended → completed`로 구현**
  현재 Worker/On-Air Player에는 `finish` 명령이 없다. 스킵은 정상 종료 확인 전 대기열·이력을 직접 바꾸는 경로이며, 화면 버튼도 클릭 이벤트를 곡 ID로 오인해 동작하지 않는다.
- [ ] **현재 곡 쓰레기통을 `discarding → discarded`로 구현**
  현재 Worker/On-Air Player에는 `discard`와 Player의 `discarded` 확인 이벤트가 없다. 기존 `stop`은 Player 완료 확인 전 transport의 곡을 비우므로, 실제 Player의 stop/unload 확인 전에는 곡을 지우지 않고 `취소 중`으로 표시해야 한다. 이력과 자동 다음 곡은 절대 발생시키지 않는다.
- [ ] **실제 OBS Player presence/lease를 상태로 사용**
  Worker는 Player presence를 control에 보내지만 현재 hook은 이를 상태로 저장하지 않는다. 따라서 UI의 control WebSocket 연결은 실제 Player 연결과 다르다. Player가 없거나 테스트 탭이 여러 개일 때도 안전한 단일 활성 Player 정책이 필요하다.
- [ ] **준비됨의 근거를 구현**
  `MR 연결됨`, iframe 생성, 광고 재생, 실제 곡 준비를 구분해야 한다. 특히 YouTube 광고/실제 타임라인을 현재 구현은 신뢰성 있게 분리하지 못한다.

### P1 — 세션·대기열·복구 신뢰성

- [ ] 종료된 세션의 토큰·로컬 자산 참조·대기열을 함께 정리하고, 401/재연결 실패 시 새 세션으로 안전하게 복구한다. 현재는 종료된 세션의 일반 WebSocket close가 1008/1011이 아니면 재연결 루프로 남을 수 있다.
- [ ] 자동 다음 곡, 대기열의 바로 재생, 재시작, 오류 재시도를 생애주기 전이 하나로 통합한다.
- [ ] 실패 곡을 완료 이력으로 넣거나 자동으로 다음 곡으로 넘기지 않는다. 재시도·재탐색·폐기 행동을 제공한다.
- [ ] 로컬 파일의 재생/재예약/재생성 시 Blob URL 수명과 R2 자산 만료를 정확히 처리한다.
- [ ] 대기열 드래그·삭제·Undo가 항목 ID 기준으로 동작하도록 정리한다.
- [ ] 화면정보 위젯이 Dashboard의 낙관적 상태가 아니라 Worker의 확정 projection을 렌더하도록 바꾼다.

### P1 — 카탈로그·AI 처리 신뢰성

- [ ] Gemini 키를 trim·검증하고, 실패 시 다음 유효 키로 재시도한다. 줄바꿈/따옴표가 섞인 환경 변수도 안전하게 처리한다.
- [ ] YouTube 플레이리스트와 대량 노래책의 제목 캐시를 일괄 조회하고, 캐시 미스만 지속 가능한 작업 큐로 처리한다. 현재의 개별 요청 + 동시성 2 + 60초 제한 구조는 대량 목록에서 지나치게 오래 걸릴 수 있다.
- [ ] Setlink 항목의 안정적 원본 ID를 보존해 목록 순서 변경으로 MR 캐시가 다른 곡에 붙지 않게 한다.
- [ ] 캐시된 MR도 최근 검증 시각과 재검증 상태를 갖게 해, 삭제/비공개/임베드 불가 MR이 곧바로 사용 가능으로 보이지 않게 한다.
- [ ] 실패한 제목 정리 항목의 재분석을 모든 입력 위치에서 같은 흐름으로 제공한다.

### P2 — 상태를 드러내는 UI/UX

- [ ] 현재 재생과 대기열에 `queued`, `preparing`, `ready`, `starting`, `finishing`, `discarding`, `failed`를 일관되게 표시한다.
- [ ] 전이 중에는 허용되지 않는 버튼을 숨기거나 비활성화하고, 이유와 다음 행동을 표시한다.
- [ ] 현재 곡 쓰레기통, 다시 시작, 바로 재생의 문구·아이콘·결과를 생애주기 정의와 일치시킨다.
- [ ] 좁은 화면과 넓은 화면 모두에서 현재 재생·대기열·곡 추가의 우선순위를 유지한다.
- [ ] 상태 전환 애니메이션은 장식이 아니라 추가·준비·전환·폐기의 인과관계를 보여 주는 용도로만 사용한다.

## 4. 다음 구현 순서

1. **상태 모델을 Worker에 이식합니다.** QueueEntry, PlaybackRun, 단일 활성 run, Player lease, 확정 projection을 먼저 만듭니다.
2. **Player 프로토콜을 맞춥니다.** `prepare`, `play`, `pause`, `seek`, `restart`, `finish`, `discard` 명령과 `ready`, `playing`, `paused`, `ended`, `discarded`, `failed` 회신을 구현합니다.
3. **Dashboard와 Widget을 projection 소비자로 바꿉니다.** UI가 명령 전송만으로 현재 곡·이력·재생 상태를 확정하지 않게 합니다.
4. **대기열과 현재 재생 UI를 phase 기반으로 바꿉니다.** 이 단계에서 쓰레기통·스킵·다시 시작을 화면에 다시 넣습니다.
5. **카탈로그·AI 대량 처리와 캐시 검증을 정리합니다.** 방송 제어의 상태 진실성이 먼저 확보된 뒤 처리합니다.

## 6. 이번 스냅샷의 배포 원칙

이번 커밋은 문서화와 안전한 UI 정리만 포함한다. 생애주기 계약을 충족하지 못하는 현재 곡 쓰레기통·스킵 동작은 임시로 추가하거나 배포하지 않는다. 해당 기능은 P0의 Player 확인 프로토콜이 준비된 뒤에만 노출한다.
