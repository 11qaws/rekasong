# 이전 재생 곡 점진 렌더 계약 — 2026-07-22

## 목적

장시간 사용으로 이전 재생 곡이 1,000개 이상 쌓여도 닫힌 이력 영역이 페이지 시작과 일반 재생 조작을 무겁게 만들지 않게 한다. 저장된 이력을 삭제하거나 임의로 축약하지 않고, 화면에 만드는 행의 수만 제한한다.

## 사용자 계약

1. 이력 영역이 닫혀 있으면 이력 행과 직접 추가 입력줄을 DOM에 만들지 않는다.
2. 처음 열면 가장 최근 100곡을 보여 준다.
3. 숨은 이력이 있으면 이전 100곡씩 더 볼 수 있다. 마지막 묶음은 남은 수만 표시한다.
4. 많이 펼친 뒤 `최근 100곡만 보기`로 즉시 접을 수 있다.
5. 이력 영역 자체를 닫으면 다음에 열 때 다시 최근 100곡부터 시작한다.
6. 전체 개수, 저장 데이터, 다시 대기열에 추가, 삭제, 드래그 재정렬은 기존과 같다.
7. 번역 문구는 semantic key로 제공한다.

## 성능 경계

- 기본 화면: 이력 행 0개.
- 이력 최초 개방: 최대 100개.
- 사용자가 명시적으로 더 보기를 누르기 전에는 100개를 넘지 않는다.
- window 계산은 원본 배열을 수정하지 않고 마지막 `limit`개만 얕게 복사한다.
- localStorage schema와 Worker/OBS protocol은 바꾸지 않는다.
- 새 timer, polling, WebSocket message, storage write를 만들지 않는다.
- 개발 서버의 `?__rekasong_history_fixture=N`은 최대 2,000개의 메모리 전용 이력을 초기 상태에 만든다. 저장된 사용자 상태를 읽거나 덮어쓰지 않으며 production build에는 query key와 생성 분기가 남지 않아야 한다.

## 검증

- 0·1·100·101·1,000곡에서 보이는 범위와 숨은 수를 단위 테스트한다.
- 1,000곡 최초 window가 최근 100곡이고 원본 순서·배열을 보존하는지 확인한다.
- 100곡씩 확장하고 전체 개수 이상으로 넘지 않는지 확인한다.
- QueuePanel이 닫힌 상태에는 window 행을 렌더하지 않고, 열 때만 렌더하며 닫을 때 limit을 초기화하는 정적 계약을 둔다.
- 개발 fixture 1,000곡으로 실제 Dashboard의 닫힘 0행·최초 개방 100행·다음 개방 200행·접기 100행을 계측한다. production bundle에 fixture query 문자열이 없는지 확인한다.
- 전체 테스트, lint, production build, OBS bundle 예산을 다시 통과해야 한다.

## 2026-07-22 실제 브라우저 계측

- 개발 fixture 1,000곡, 데스크톱 기본 화면: 이력 행 0개.
- 최초 개방과 5회 반복 개방: 항상 최근 100개만 생성. 자동화 왕복을 포함한 개방 시간은 317~374ms였다.
- 닫기 5회: 매번 이력 행이 0개로 제거됨. 자동화 왕복을 포함한 닫기 시간은 323~339ms였다.
- `이전 100곡 더 보기`: 200개로 확장되고 첫 행은 0801, 마지막 행은 1000이었다. `최근 100곡만 보기`는 다시 100개로 줄였다.
- 390×844 모바일 viewport: 문서 가로 폭과 scroll 폭이 모두 380px로 같아 가로 넘침이 없었다.
- 모바일에서 이력을 닫으면 전체 DOM 요소가 2,530개에서 120개로 줄고, 이력 행은 100개에서 0개가 됐다.
- 브라우저 오류 로그 0건. production build에서 fixture query·제목·작성자 문자열은 0건이었다.
