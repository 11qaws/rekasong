# Speaker 탭 상태 수명 계약 — 2026-07-23

## 1. 사용자 기준 결론

Speaker는 한 브라우저 안에서도 각 탭이 서로 독립된 일반 음악 플레이어다.
한 탭에서 곡을 대기열에 넣거나 자동 다음 곡을 켠 일이 다른 탭의 재생 준비 상태를
만들거나 바꾸면 안 된다. 반면 이전 재생곡과 가져온 노래책은 사용자의 라이브러리이므로
같은 브라우저의 탭에서 함께 보인다.

## 2. 상태별 소유자와 수명

| 상태 | 소유자 | 저장 위치 | 새로고침 | 다른 탭 |
|---|---|---|---|---|
| 현재 곡·재생 run·실제 미디어 요소 | 현재 탭 | 메모리 | 정지하고 비움 | 절대 전달하지 않음 |
| 대기열·자동 다음 곡 | 현재 탭 | `sessionStorage` | 같은 탭에서 복원 | 전달하지 않음 |
| 이전 재생곡·노래책·MR 연결·사용 환경설정 | 브라우저 사용자 | `localStorage` | 복원 | 공유 |
| 로컬 파일의 실제 `blob:` URL | 현재 페이지 | 메모리 | 만료 안내로 전환 | 절대 저장·전달하지 않음 |
| 언어 | 브라우저 사용자 | 별도 locale key | 복원 | 같은 브라우저에서 공유 |

## 3. 허용 전이

| 현재 상태 + 이벤트 | 결과 |
|---|---|
| 새 탭 열기 | 현재 곡 없음, 대기열 없음, 자동 다음 곡 꺼짐, 공유 이력·노래책만 로드 |
| 이 탭에서 대기열 변경 | 이 탭의 session 상태만 변경 |
| 이 탭에서 자동 다음 곡 변경 | 이 탭의 session 상태만 변경 |
| 이 탭에서 곡 완료 | 이 탭의 현재 run 종료 + 공유 이력 갱신 |
| 다른 탭에서 이력·노래책 변경 | 이 탭의 현재 곡·대기열·자동 다음 곡을 보존한 채 라이브러리만 병합 |
| 같은 탭 새로고침 | 현재 재생은 자동 재개하지 않고, 대기열·자동 다음 곡만 복원 |
| 구버전 상태 최초 로드 | 구 공유 대기열을 이 탭 session으로 한 번만 이관 |

## 4. 하위 호환과 배포 전이

- 구 key `karaoke_app_state`는 읽기 전용 이관 원본으로만 사용한다.
- 새 공유 key가 아직 없고 이 탭 session도 없을 때만 구 대기열을 이 탭으로 가져온다.
- 새 앱은 구 key를 덮어쓰지 않는다. 배포 중 이미 열려 있는 구버전 탭의 대기열을
  새 탭이 지우는 혼합 버전 사고를 막기 위해서다.
- 한 번 만들어진 탭 session은 빈 대기열도 명시적인 사용자 상태다. 이후 구 공유
  대기열이 다시 스며들지 않는다.
- 브라우저의 “탭 복제”가 sessionStorage를 복사하는 경우 초기 목록이 복제될 수 있지만,
  복제 직후부터 두 탭의 변경과 재생 run은 완전히 독립적이며 자동 재생하지 않는다.

## 5. 불변식

1. storage event는 다른 탭의 `queue`, `autoPlayNext`, `currentEntry`, `active`를 바꾸지 않는다.
2. 공유 payload에는 현재 곡·active run·대기열·자동 다음 곡을 싣지 않는다.
3. 탭 payload에는 현재 곡·active run·실제 Blob URL을 싣지 않는다.
4. 새 탭은 다른 탭의 준비 중·준비됨·재생 중 표시를 만들지 않는다.
5. 새로고침 복원은 재생 허가가 아니다. 사용자가 다시 재생을 눌러야 실제 출력이 시작된다.
6. Speaker의 이 상태 분리는 OBS의 단일 방송 출력 lease를 완화하지 않는다.

## 6. 검증 시나리오

1. A와 B를 열고 A에서만 이력 곡을 대기열에 넣는다. A는 1곡, B는 0곡이어야 한다.
2. A에서만 자동 다음 곡을 켠다. A만 켜지고 B는 꺼져 있어야 한다.
3. 두 탭을 새로고침한다. 각 탭의 대기열·자동 다음 곡이 자기 값으로 돌아와야 한다.
4. C를 새로 연다. C는 빈 대기열·자동 다음 곡 꺼짐이면서 공유 이력은 보여야 한다.
5. 어느 탭에서도 현재 곡은 새로고침 뒤 자동 재생되지 않아야 한다.
6. 로컬 파일 항목은 새로고침 뒤 실제 Blob 대신 파일 재선택 안내가 나와야 한다.
