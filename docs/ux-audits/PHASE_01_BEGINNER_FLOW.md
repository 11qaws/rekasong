# Phase 01 — 초심자 방송 흐름 UX 감사 (30문항)

## 범위와 성공 시나리오

대상 사용자는 처음 Rekasong을 연 스트리머다. 성공은 **곡을 찾고 → 방송에 표시될 정보를 확인하고 → 재생과 OBS 위젯 표시를 확신하는 것**이다. 이 과정에 없는 설정은 기본 화면의 판단 부담을 늘리지 않아야 한다.

판정 표기: `통과` / `보완` / `위험`

| # | 감사 질문 | 현황 판정 | Phase 01 결정 |
|---:|---|---|---|
| 1 | 첫 화면만 보고 앱의 목적을 알 수 있는가? | 보완 | 헤더 설명을 방송용 노래 검색·재생·OBS 제어로 유지한다. |
| 2 | 첫 행동인 검색이 1단계로 명확한가? | 보완 | 검색 패널에 `1단계 노래 찾기` 제목을 추가한다. |
| 3 | 검색창이 곡명·가수·URL을 모두 받는다는 점이 보이는가? | 통과 | 현재 플레이스홀더를 유지한다. |
| 4 | 검색 결과를 누르면 다음 단계로 갔음을 알 수 있는가? | 위험 | 스테이징 전환 토스트와 단계 안내를 제공한다. |
| 5 | 유튜브 URL을 붙여넣으면 불필요한 검색 단계를 거치지 않는가? | 통과 | 영상 ID·URL 직접 스테이징을 유지한다. |
| 6 | 로컬 파일은 클릭과 드롭 모두 가능한가? | 통과 | 전체 드롭 영역·형식·용량 검증을 유지한다. |
| 7 | 스테이징에서 곡명과 가수를 고친 뒤 무엇을 눌러야 하는가? | 통과 | 상황별 `즉시 재생`/`대기열 추가` CTA를 유지한다. |
| 8 | 스테이징을 비우면 진행 중 AI 분석도 함께 정리되는가? | 위험 | 비우기 동작에서 AI 요청을 취소하고 피드백을 준다. |
| 9 | AI가 사용자가 직접 수정한 제목을 덮어쓰지 않는가? | 통과 | 수동 편집 보호와 요청 식별자를 유지한다. |
| 10 | AI 실패·지연 시 사용자가 막히지 않는가? | 통과 | 입력을 열어 둔 채 직접 입력 가능 상태를 표시한다. |
| 11 | 곡 선택 때 기존 스테이징이 바뀌었음을 알 수 있는가? | 보완 | 새 선택 알림을 표시한다. |
| 12 | 초기 화면에서 비어 있는 2단계가 공간을 차지하지 않는가? | 통과 | 스테이징 전에는 검색·방송 제어 2단 구성을 유지한다. |
| 13 | 스테이징이 열렸을 때 순서가 좌→우로 읽히는가? | 보완 | 세 패널 모두 단계 번호와 짧은 행동명으로 통일한다. |
| 14 | 정보 확인보다 영상 미리보기가 먼저 시선을 빼앗지 않는가? | 통과 | 입력·CTA를 미리보기보다 위에 둔다. |
| 15 | 현재 재생 중인지 방송자가 즉시 알 수 있는가? | 통과 | ON AIR 배지와 재생/일시정지 제어를 유지한다. |
| 16 | 잘못된 곡을 즉시 멈출 수 있는가? | 통과 | 2단계 확인형 긴급 정지를 유지한다. |
| 17 | 자동 다음 곡의 상태가 화면·실행 로직·위젯에서 일치하는가? | 통과 | `autoPlayNext` 동기화 상태 하나만 사용한다. |
| 18 | 대기열의 순서를 바꾸고 실수를 되돌릴 수 있는가? | 보완 | 드래그 정렬과 이력 재예약은 있으나 삭제 Undo는 다음 회차 후보로 둔다. |
| 19 | 이전 재생 곡을 위젯에서 제거하거나 다시 예약할 수 있는가? | 통과 | 삭제 확인과 맨 앞 재예약을 유지한다. |
| 20 | 위젯 주소 복사는 매 방송마다 해야 한다고 오해하지 않는가? | 통과 | 위젯 설정은 접힌 고급 영역으로 유지한다. |
| 21 | 위젯이 늦게 열려도 현재 상태를 보이는가? | 통과 | 방별 저장 상태 복원을 유지한다. |
| 22 | 동기화가 실패했을 때 실제 방송 상태와 다르게 믿게 하지 않는가? | 보완 | 네트워크 동기화 실패 상태 표시는 다음 회차 후보다. |
| 23 | 멜로밍·Setlink이 실제 연동이라고 오해될 여지가 없는가? | 통과 | 데모 데이터 배너를 유지한다. |
| 24 | 핵심 검색 흐름에 준비되지 않은 연동이 과도하게 끼어들지 않는가? | 보완 | 탭은 유지하되 데모 상태와 실제 제약을 명확히 표시한다. |
| 25 | `실시간 송출 관리` 같은 용어가 초심자에게 바로 이해되는가? | 보완 | `방송 제어`처럼 행동 중심 문구로 교체한다. |
| 26 | 빈 상태가 다음 행동을 지시하는가? | 통과 | 검색·재생·대기열의 CTA형 빈 상태를 유지한다. |
| 27 | 긴 곡명·가수명·태그가 레이아웃을 깨지 않는가? | 통과 | 말줄임·줄바꿈 처리 현행을 유지하고 실제 데이터로 후속 검증한다. |
| 28 | 키보드 단축키가 입력 중에는 오작동하지 않는가? | 통과 | 입력 요소에서는 전역 단축키를 무시한다. |
| 29 | 브라우저 새로고침 후 핵심 설정이 불필요하게 초기화되지 않는가? | 통과 | 방 ID·볼륨·자동재생·연동 탭 상태를 유지한다. |
| 30 | 정적 배포와 로컬/Cloudflare 동기화의 차이가 사용자 기대를 깨지 않는가? | 보완 | 배포 제약을 개발 로그에 기록했으며, 화면 내 연결 상태 표시는 다음 회차 후보다. |

## Phase 01 구현 우선순위

1. **P0 — 상태 피드백:** 곡 선택과 스테이징 취소처럼 암묵적으로 바뀌는 상태를 토스트로 알린다.
2. **P1 — 단계 인지:** 검색과 방송 제어의 제목을 행동 중심 1·3단계 문구로 통일한다.
3. **P1 — 안전한 취소:** 스테이징 취소는 AI 요청까지 취소한다.
