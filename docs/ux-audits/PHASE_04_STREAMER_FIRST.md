# UX 감사 Phase 04 — 노래 스트리머 실전 우선

기준: 이 앱의 최종 사용자는 방송 중인 스트리머다. 테마나 기능 수보다, 익숙한 방식으로 다음 곡을 안전하게 이어가고 실수를 즉시 되돌릴 수 있는지가 우선이다.

## 30문항 판정

| # | 질문 | 판정 | 조치 |
|---:|---|---|---|
| 1 | 처음 온 스트리머가 다음 행동을 알 수 있는가? | 통과 | 단계형 첫 화면 유지 |
| 2 | 노래를 넣기 전 재생 화면을 찾을 필요가 없는가? | 통과 | 검색·대기열·재생을 한 흐름으로 유지 |
| 3 | 방송 중 대기열에서 잘못 지운 곡을 바로 복구할 수 있는가? | 미흡 | 제거 후 되돌리기 제공 |
| 4 | 긴급 정지가 무엇을 멈추는지 누르기 전에 알 수 있는가? | 미흡 | 2차 버튼에 현재 곡·대기열 범위 명시 |
| 5 | 긴급 정지 후 상태 변화를 확인할 수 있는가? | 미흡 | 완료 알림 제공 |
| 6 | 평소에는 위험 제어가 실수로 실행되기 어려운가? | 통과 | 2단계 확인 유지 |
| 7 | 방송 중 자동 다음 곡 옵션이 화면을 어지럽히지 않는가? | 통과 | 접힌 재생 옵션 유지 |
| 8 | 자동 다음 곡은 사용자가 켰을 때만 동작하는가? | 통과 | 기본값 꺼짐 |
| 9 | 현재 재생 곡을 빠르게 확인할 수 있는가? | 통과 | 재생 패널 상단 노출 |
| 10 | 대기열 순서가 방송 진행 순서로 읽히는가? | 통과 | 목록 순서 유지 |
| 11 | 로컬 파일을 넣었을 때 재생 가능한 파일인지 바로 알 수 있는가? | 통과 | 형식·용량 검사와 알림 유지 |
| 12 | URL을 붙여넣으면 별도 설명 없이 검색할 수 있는가? | 통과 | URL/영상 ID 인식 유지 |
| 13 | AI 제목 추출을 취소해도 수동 입력이 보존되는가? | 통과 | 취소·수동 편집 보호 유지 |
| 14 | 연결되지 않은 외부 음원 서비스가 실제 서비스처럼 보이지 않는가? | 통과 | 데모 표기 유지 |
| 15 | OBS 위젯 주소를 복사한 뒤 다음 행동을 알 수 있는가? | 통과 | 최초 1회 설정 설명 유지 |
| 16 | 위젯이 실제로 수신 중인지 근거 없이 ‘연결됨’으로 보이지 않는가? | 통과 | 수신 확인 전에는 연결 상태를 단정하지 않음 |
| 17 | 검색 실패가 방송 진행을 막지 않는가? | 통과 | 직접 URL·로컬 파일 경로 유지 |
| 18 | 한 곡을 고른 뒤 대기열에 넣는 결과가 즉시 보이는가? | 통과 | 단계·알림 유지 |
| 19 | 방송 중 알림이 핵심 제어를 가리지 않는가? | 통과 | 짧은 비차단 알림 유지 |
| 20 | 알림 안의 복구 동작이 키보드로도 실행 가능한가? | 통과 | 버튼으로 제공 |
| 21 | 되돌리기가 다른 대기열 변경을 덮어쓰지 않는가? | 통과 | 동일 곡이 없을 때만 원래 위치에 삽입 |
| 22 | 곡을 여러 번 삭제해도 각각의 복구 의도를 구분할 수 있는가? | 통과 | 곡 제목을 포함한 알림 제공 |
| 23 | 공개 방송에서 민감한 API 키가 화면이나 로그에 노출되지 않는가? | 통과 | 서버 측 환경 변수만 사용 |
| 24 | 음원 중복 출력 위험을 피하는 구조가 유지되는가? | 통과 | YouTube 오디오는 앱 내 재생, OBS는 메타데이터 위젯만 사용 |
| 25 | 재생 중 새 검색을 해도 현재 곡이 갑자기 바뀌지 않는가? | 통과 | 스테이징 후 명시적 재생 유지 |
| 26 | 위젯 연결 불확실성이 방송자에게 오판을 유도하지 않는가? | 통과 | 실제 수신 확인 전까지 상태 배지 추가 보류 |
| 27 | 기능 이름이 방송자가 쓰는 언어와 맞는가? | 통과 | 대기열·현재 곡·긴급 정지 사용 |
| 28 | 첫 방송 전 설정이 매번 방송 중에 노출되지 않는가? | 통과 | OBS 안내 접기 유지 |
| 29 | 오류·취소·완료가 서로 다른 의미로 구분되는가? | 통과 | 알림 유형 유지 |
| 30 | 꾸밈보다 방송 흐름이 더 빠르고 익숙한가? | 진행 | Phase 04의 복구·안전 제어를 우선 반영 |

## 이번 반영의 경계

위젯 수신 확인은 현재 동기화 채널에 수신 응답이 없어 실제 연결을 검증할 수 없다. 따라서 ‘연결됨’ 같은 상태 표시는 추가하지 않는다. 향후 위젯 heartbeat와 인증된 확인 응답을 설계한 뒤에만 상태를 표시한다.
